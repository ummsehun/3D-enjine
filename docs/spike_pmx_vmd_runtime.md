# PMX/VMD Runtime Spike (v7.2)

## 결론
- **No-Go (현재 분기 기준)**  
  PMX/VMD 런타임 직접 파싱/렌더 전환은 지금 품질/성능 이슈를 해결하는 최단 경로가 아니다.
- **권장 경로**  
  `PMX/VMD -> Blender/MMD Tools -> GLB` 전처리 파이프라인 유지 + 런타임은 GLB 품질/안정화 집중.

## 이유
- PMX/VMD 런타임 전환 시, 최소 구현 범위가 단순 파싱을 넘어선다.
- 특히 `좌표계 변환`, `보간`, `IK/물리`, `재질 표현`, `도구 호환`에서 위험도가 높다.
- 현재 병목의 핵심은 포맷 자체보다 `터미널 출력 경로`, `복구 상태머신`, `톤매핑/색공간 정합`이다.

## 기술 리스크 요약

### 1) 좌표계/단위/회전 변환
- MMD(VMD)와 GLTF/엔진 좌표계 차이로 카메라/모션 역방향, 지면 관통, 스케일 붕괴 위험.
- 필수: 축 반전, 단위 스케일, 회전 부호 및 오일러->쿼터니언 표준화.

### 2) 보간/애니메이션 품질
- VMD 보간(Bezier)과 런타임 샘플러 차이로 동작 느낌이 달라질 수 있음.
- 얼굴/손 디테일은 키프레임 밀도, morph 적용 순서, 보간 정확도 영향이 큼.

### 3) IK/물리/의상
- MMD의 강점인 IK/물리(스커트/헤어/보조본)를 런타임에서 동일하게 재현하려면 별도 엔진 수준 구현 필요.
- 비용 대비 효과가 낮고, 안정성 리스크가 큼.

### 4) 재질/색 정확도
- PMX 재질 시스템과 glTF PBR 간 완전 대응이 어려워, 전환 시 오히려 색/광택 일관성이 떨어질 수 있음.
- 현재 이슈(반전/바램)는 포맷 교체보다 색공간/샘플링/톤매핑 개선이 우선.

## 개발비/유지비 비교
- **GLB 유지**: 낮음 ~ 중간  
  - 장점: 툴체인 성숙, 자료 많음, 런타임 단순화.
- **PMX/VMD 런타임 전환**: 높음 ~ 매우 높음  
  - 장점: 원본 충실도 잠재력.
  - 단점: 구현 범위 급증, 디버깅 난이도 급상승, 성능/안정성 리스크 큼.

## 조건부 Go 기준 (차기 분기)
아래를 모두 만족할 때만 PMX/VMD 런타임 전환을 재검토:

1. KittyHQ/텍스트 경로 모두에서 10분 안정 실행(패닉/고착 0).
2. 색 반전/바램 회귀율이 낮고, 모델 가시성 자동복구가 신뢰 가능.
3. GLB 기반으로 목표 연출(카메라, 동기화, stage sub, 근접/원거리 품질)이 수용 기준 충족.
4. PMX/VMD 런타임 전환에 투입 가능한 별도 스프린트 확보(최소 2~3주 이상).

## 실행 권장안
- 이번 분기: GLB 경로 고도화 지속.
- 차기 분기: PMX/VMD는 **Spike only**를 확장(샘플 파서+카메라/모션 정합 테스트)하고, 본 전환은 조건부.
